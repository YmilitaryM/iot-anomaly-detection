统一检测方案：LSTM-Autoencoder

一个模型同时解决时序模式 + 多维关联 + 上下文感知

STL + IF 组合的本质缺陷

STL 残差检测和 Isolation Forest 是各自为战的两个独立模型：

• STL 在时域上逐个传感器分解，不知道其他传感器在发生什么

• IF 在特征空间中看离群点，不知道数据点之间的先后顺序

• 两个模型的分数靠融合器人工加权拼在一起，没有内在的有机联系

根本原因：时序和空间（多传感器）是分离处理的，但真实异常往往同时涉及两者。

LSTM-Autoencoder 如何统一解决

核心原理
输入: 滑动窗口 [T-n, ..., T] × 所有传感器 → 一整个多维时序片段
编码器 (LSTM)
逐步压缩时序窗口
→ 固定长度的隐向量
包含了"这段时间内所有传感器
的正常行为模式"
→
解码器 (LSTM)
从隐向量重建整个时序窗口
→ 尝试还原输入
如果输入是"正常模式"
重建会很准确
anomaly_score = reconstruction_error = ||输入 - 重建||₂
重建误差大 → 模型"没见过这种模式" → 异常

更新后的务实路线

Phase 1 (第1-2周): 基础管线
• MQTT接入 + 数据清洗 + 时序库 + 硬边界检测
• 基础统计规则 (变化率、死区、简单阈值)
• 告警去重 & 通知 & 基础仪表盘
• 在此期间积累正常数据，作为 Phase 2 的训练集
Phase 2 (第3-4周): LSTM-Autoencoder 上线
• 用 Phase 1 积累的正常数据训练模型
• 替代所有统计规则，单一模型输出 anomaly_score
• 异常可定位: 哪个传感器在哪个时间步重建误差最大
• 保留硬边界作为安全兜底
Phase 3 (第5-6周): 反馈闭环 + 模型进化
• 运维确认/拒绝 → 积累标注
• 定期重训练 (在线增量 或 滑动窗口重训)
• 概念漂移监控 → 触发模型更新
Phase 4+: 按需扩展
• 如果某类场景 LSTM-AE 反复误报 → 补充规则/变点检测
• Transformer替代LSTM (数据量足够大时)
• 依然是插件式架构，但不预设复杂度

需要正视的代价

相比 STL+IF 的提升

  • 架构从"两个模型+融合器"变成"一个模型"，运维复杂度降一半
  • 天然处理时序+多维+上下文，不需要额外的上下文感知层
  • 缓慢漂移能检测（重建误差趋势上升本身就是信号）
  • 异常定位更直观：直接看哪个传感器在哪个时间步重建最差

新增的挑战

  • 需要 Phase 1 积累正常数据（最少几千个时间窗口）才能训练
  • 训练和推理比纯统计方法重（但远轻于 Transformer）
  • 超参数需要调试：窗口大小、隐层维度、层数
  • 概念漂移时需要重训练策略
  • 比规则更难解释"为什么异常"（但可以用逐传感器重建误差来近似解释）
