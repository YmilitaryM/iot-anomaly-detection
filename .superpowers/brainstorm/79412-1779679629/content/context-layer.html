上下文突变检测 — 当前设计的缺口与补充

三种上下文异常，当前设计只覆盖了一部分

上下文异常的三种类型

类型 例子 当前设计覆盖？
时间上下文 40°C 白天正常，夜间异常。周末 vs 工作日用电模式不同。 部分(STL季节分量可覆盖固定周期，但不能处理非固定模式的上下文)
工况上下文 设备在启动/满载/空载/停机时正常值差异很大。60°C满载正常，空载就是故障。 缺失 — 当前没有工况识别和条件归一化
关联上下文 温度高+电流低=异常(散热故障)。温度高+电流高=正常(负载高)。单个传感器看都正常，组合才异常。 部分(LSTM和邻近度检测器可捕捉，但不够显式)

解决方案：在Pipeline中增加 "上下文感知层"

改进后的 Pipeline — 在 Stage 1 和 Stage 2 之间插入 Context Layer
Stage 1: 预处理 & 时序分解 [不变]
🆕 Stage 1.5: 上下文感知层

1. 工况识别 (Operating Mode Detection)
• 用 DBSCAN 对多维工况特征(RPM、负载率、功率、档位等)做在线聚类
• 每个聚类 = 一个工况模式，实时判定当前处于哪个工况
• 新工况自动发现 → 标记为 "未知模式" 触发人工标注
• 输出: mode_id, mode_confidence

2. 条件归一化 (Conditional Normalization)
• 在每个工况内分别计算基线(均值/方差/分位数)
• 传感器原始值 → mode_relative_zscore = (value - mode_mean) / mode_std
• 将 "相对工况的偏差" 作为检测器输入，替代原始值
• 效果: 60°C满载 → zscore≈0(正常)，60°C空载 → zscore>>3(异常)

3. 上下文特征注入
• 时间编码: hour_sin, hour_cos, day_of_week, is_holiday
• 工况编码: mode_id → embedding vector
• 关联快照: 同时刻其他关联传感器的残差
• 这些特征直接拼接到 Stage 2 的特征向量中
Stage 2: 特征工程 [增强: 接收上下文特征]
Stage 3: 多策略检测 [增强: 检测器输入从 raw value 变成 mode_relative_zscore]

为什么这个位置是关键设计选择

放在 Stage 1 和 2 之间

  • 时序分解已剥离周期，上下文层处理的是剩下的 "模式切换" 问题
  • 条件归一化后的值直接喂给所有5个检测器，无需每个检测器自己做上下文处理
  • 新增传感器类型只需配置哪些字段是 "工况特征"

挑战

  • DBSCAN 在线聚类需要滑动窗口 + 增量更新，不能纯批处理
  • 工况标注需要运维配合（但可以先纯无监督聚类，逐步标注）
  • 低频传感器可能没有足够维度做工况识别，需要降级策略
